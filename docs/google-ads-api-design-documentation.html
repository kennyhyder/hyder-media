1. Executive Summary

Hyder Media Client Dashboard is a read-only reporting and analytics platform that aggregates Google Ads performance data for marketing agency clients. The tool provides:

  • Centralized dashboard for clients to view their Google Ads performance
  • Automated daily data synchronization from Google Ads API
  • Historical trend analysis and performance visualization
  • Multi-account management through MCC (Manager Account) integration
Key Point: This tool is strictly read-only. We do not create, modify, or delete any campaigns, ad groups, keywords, or other Google Ads entities. We only retrieve performance metrics for reporting purposes.

2. Tool Overview

2.1 Purpose

The Hyder Media Client Dashboard serves as a centralized reporting portal where agency clients can log in to view their Google Ads campaign performance. Instead of requiring clients to navigate the Google Ads interface directly, they access a simplified, branded dashboard that presents only the metrics relevant to their business goals.

2.2 Target Users

User Type Description Access Level
Agency Administrators Hyder Media team members who manage client accounts Full access to all client data
Client Users Business owners/marketing managers at client companies Access to their own account data only

2.3 Core Features

  • Performance Dashboard: Visual overview of key metrics (spend, clicks, conversions, ROAS)
  • Campaign Breakdown: Detailed performance by campaign, ad group, and keyword
  • Search Term Analysis: Insights into actual search queries triggering ads
  • Historical Trends: Performance data over time with comparison periods
  • Automated Reporting: Daily data synchronization without manual intervention

4. Google Ads API Usage

4.1 API Services Used

Read-Only Operations Only: All API operations are strictly read-only. We use GoogleAdsService.Search and GoogleAdsService.SearchStream to retrieve data. We do not use any mutate operations.
Service Method Purpose
CustomerService ListAccessibleCustomers Discover client accounts under MCC
GoogleAdsService Search Query campaign, ad group, keyword data
GoogleAdsService SearchStream Stream large result sets efficiently

4.2 Resources Accessed (Read-Only)

Resource Data Retrieved Frequency
customer Account name, currency, timezone On connection
customer_client List of accounts under MCC On connection
campaign Campaign names, status, budgets Daily sync
ad_group Ad group names, status, bids Daily sync
ad_group_criterion (keywords) Keywords, match types, status Daily sync
search_term_view Actual search queries Daily sync
metrics Impressions, clicks, cost, conversions Daily sync

4.3 Sample GAQL Queries

Campaign Performance Query:

SELECT campaign.id, campaign.name, campaign.status, campaign.advertising_channel_type, metrics.impressions, metrics.clicks, metrics.cost_micros, metrics.conversions, metrics.conversions_value FROM campaign WHERE segments.date DURING LAST_30_DAYS ORDER BY metrics.cost_micros DESC

Search Terms Query:

SELECT search_term_view.search_term, campaign.name, ad_group.name, metrics.impressions, metrics.clicks, metrics.cost_micros, metrics.conversions FROM search_term_view WHERE segments.date DURING LAST_7_DAYS ORDER BY metrics.impressions DESC LIMIT 1000

7. Security & Compliance

Security First: We implement industry-standard security practices to protect Google Ads data and OAuth credentials.

7.1 Authentication & Authorization

Security Measure Implementation
OAuth 2.0 Standard Google OAuth 2.0 flow with PKCE
Token Storage Encrypted at rest in Supabase (AES-256)
Token Refresh Automatic refresh before expiry
Access Scope Minimum required: https://www.googleapis.com/auth/adwords
Client Authentication Email/password with Supabase Auth (bcrypt hashing)

7.2 Data Protection

Protection Layer Details
Transport Security TLS 1.3 for all communications
Database Encryption Supabase PostgreSQL with encryption at rest
Row-Level Security PostgreSQL RLS policies ensure clients only see their own data
API Security Environment variables for secrets, never exposed to client
Access Logging All API calls logged with timestamps and user context

7.3 Data Retention & Privacy

  • Data Minimization: We only store data necessary for reporting
  • Retention Period: Performance data retained for 13 months (for YoY comparison)
  • Data Deletion: Client data deleted upon request or account termination
  • No Data Sharing: Client data is never shared with third parties
  • Compliance: GDPR and CCPA compliant data handling

7.4 Infrastructure Security

  • Vercel Platform: SOC 2 Type II certified infrastructure
  • Supabase: SOC 2 Type II certified, hosted on AWS
  • No Server Access: Serverless architecture eliminates server management risks
  • Automatic Updates: Platform handles security patches automatically

8. Standard Access Justification

8.1 Current Usage & Growth

As a digital marketing agency, Hyder Media manages Google Ads campaigns for multiple clients. Our current and projected usage requires Standard Access:

Metric Current Projected (12 months)
Client Accounts Managed 15+ 50+
Daily Sync Operations ~10,000 ~50,000+
Data Points per Account ~500-1000 ~500-1000

8.2 Why Basic Access is Insufficient

The 15,000 daily operation limit of Basic Access becomes constraining as we scale:

  • Multi-Account Syncing: Each client account requires multiple queries (campaigns, ad groups, keywords, search terms, metrics)
  • Historical Data: Initial sync and backfill operations require significant API calls
  • Real-Time Dashboard: Users expect up-to-date data, requiring more frequent syncs
  • Growth Trajectory: Adding new clients would quickly exceed Basic Access limits

8.3 Operations Breakdown

For each client account, a full daily sync requires approximately:

Query Type Approx. Operations
Campaign data + metrics 50-100
Ad group data + metrics 100-300
Keyword data + metrics 200-500
Search term data 100-200
Total per account 450-1,100

With 50 client accounts: 22,500 - 55,000 daily operations (exceeds Basic Access limit)

8.4 Commitment to API Policies

We commit to:

  • Adhering to all Google Ads API Terms of Service
  • Implementing Required Minimum Functionality (RMF) as applicable
  • Maintaining read-only access (no mutate operations)
  • Respecting rate limits and implementing exponential backoff
  • Keeping our integration updated with API version changes
